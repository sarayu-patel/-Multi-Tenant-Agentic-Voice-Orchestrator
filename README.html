Multi-Tenant
Voice Orchestrator

A cloud-native SaaS system where multiple real-estate companies (tenants) can run AI-powered outbound calling campaigns to qualify their leads — fully automated, from trigger to transcript analysis.

FastAPI LangGraph Vapi AI MongoDB OpenAI GPT-4o-mini Google Gemini GCP Cloud Run Docker
Big Picture

What this system does

A company manager clicks one button. The system finds every lead that hasn't been called yet, calls each of them using an AI voice agent, listens to the conversation, and automatically updates each lead's status — all without human involvement.

🏠 Sunset Realty wants to call their pending leads
1
Manager hits POST /api/campaigns/{company_id}/trigger. The system fetches all leads for Sunset Realty that still have status PENDING.
2
For each lead, our server calls Vapi's REST API to initiate an outbound phone call. The AI agent's system prompt is built dynamically from Sunset Realty's profile — it mentions houses, not apartments.
3
Each lead's status immediately updates to CALL_INITIATED in MongoDB. The Vapi call ID is saved so we can match the webhook later.
4
Alice's phone rings. She speaks with the AI for 2 minutes. The call ends.
5
Vapi sends an end-of-call-report webhook to our POST /api/webhooks/vapi endpoint with the full transcript and summary.
6
LangGraph's evaluation graph runs — it sends the transcript to OpenAI GPT-4o-mini, which reads the conversation and returns a verdict: QUALIFIED.
7
Alice's lead status is updated to QUALIFIED in MongoDB. The call log is saved with the transcript and reasoning.
Specification

Functional Requirements

What the system must do.

  • FR-1Support multiple companies (tenants) each with their own leads, call history, and AI prompt configuration. One tenant's data is completely isolated from another's.
  • FR-2Allow leads to be created with a name, phone number, and company association. Each lead starts in PENDING state.
  • FR-3Trigger outbound voice calls to all PENDING leads of a company via Vapi AI's REST API in a single API call.
  • FR-4Build each AI call's system prompt dynamically from the company's prompt_instructions field so the AI sounds specific to that company.
  • FR-5Receive Vapi's end-of-call webhook, extract the transcript, and route it to the LangGraph evaluation pipeline.
  • FR-6Use an LLM (OpenAI GPT-4o-mini, Gemini as fallback) to classify each call as QUALIFIED, NOT_INTERESTED, or NEEDS_REVIEW.
  • FR-7Update the lead's status and save the transcript + LLM reasoning in the call log collection.
  • FR-8Pre-seed the database with 2 mock companies and 3 leads each on first startup.
  • FR-9Expose REST endpoints to list companies, list leads (filtered by company), and manually override a lead's status.

Non-Functional Requirements

How the system must behave beyond core features.

  • NFR-1Resilience: One lead failing to call (bad number, Vapi error) must not block the other leads in the same campaign. Errors are captured per-lead and the rest continue.
  • NFR-2LLM Fallback: If OpenAI is unavailable, the system automatically retries with Google Gemini. If both fail, the lead is flagged NEEDS_REVIEW instead of crashing.
  • NFR-3Graceful Webhooks: Empty body pings from Vapi (connectivity checks) return HTTP 200 without errors. Non-JSON bodies are logged and ignored.
  • NFR-4Human-in-the-loop: When the LLM is uncertain about a call outcome (ambiguous transcript, very short call), it returns NEEDS_REVIEW rather than guessing — a human agent should follow up.
  • NFR-5Idempotent Seeding: The seed function checks if data already exists before inserting — restarting the server never creates duplicate companies or leads.
  • NFR-6No secret leaks: All credentials (Vapi key, OpenAI key, MongoDB URI) are loaded from environment variables, never hardcoded.
  • NFR-7CORS open for development: The API allows all origins so a frontend or Postman can call it without headers issues during local development.
  • NFR-8Single DB connection: MongoDB client is created once at startup and reused — not reconnected per request.
  • NFR-9Containerizable: Multi-stage Dockerfile keeps the image lean. Cloud Run compatible (reads PORT from environment).
Design

System Architecture

Three separate concerns, cleanly separated: the API layer handles HTTP, the agent layer orchestrates logic, and the service layer talks to external systems.

🌐
API Layer (FastAPI)

Handles HTTP requests/responses. Validates inputs. Routes to the right graph or DB query. Four router files — one per domain.

🧠
Agent Layer (LangGraph)

Two stateful graphs. Campaign graph dispatches calls. Evaluation graph processes transcripts. Each node does exactly one thing.

📞
Service Layer (Vapi)

One thin wrapper around Vapi's REST API. Builds the dynamic prompt, assembles the payload, handles errors explicitly.

🗄️
Data Layer (MongoDB)

Three collections: companies, leads, call_logs. Async access via Motor. One connection shared across the entire app lifetime.

Data Layer

Database Design

MongoDB with three collections. Multi-tenancy is enforced by company_id on every lead and call log — all queries filter by it.

Collection: companies

One document per tenant (real-estate agency). The prompt_instructions field is what makes each tenant's AI calls sound different.

{
  "_id": ObjectId("6a377fa8aed11f313e1df9fe"),
  "name": "Sunset Realty",
  "prompt_instructions": "You work for Sunset Realty, a premium home-buying agency.
    Your job is to find out if the lead is looking to purchase a home,
    what their budget is, and which neighborhoods they prefer.",
  "created_at": ISODate("2024-01-15T10:00:00Z")
}

Collection: leads

One document per potential customer. The status field tracks exactly where in the calling pipeline this lead sits. vapi_call_id is the foreign key that links to Vapi's call when it happens.

{
  "_id": ObjectId("6a378113dceb6bddaaa5738b"),
  "company_id": "6a377fa8aed11f313e1df9fe",  // ties to companies._id
  "name": "Alice Johnson",
  "phone": "+14085550101",              // E.164 format required by Vapi
  "status": "QUALIFIED",               // updated by LangGraph after call
  "vapi_call_id": "019ee8d9-cb3a-700f...", // saved when call is initiated
  "created_at": ISODate("2024-01-15T10:00:00Z"),
  "updated_at": ISODate("2024-01-15T10:15:00Z")
}

Collection: call_logs

One document per call attempt. Created when the call is dispatched (transcript is null at first), then filled in when the webhook arrives.

Agentic Orchestration

LangGraph Flows

LangGraph builds a stateful computation graph — a series of functions (nodes) connected by edges. Each node receives the current state, does one thing, and returns an updated state. We have two completely separate graphs for two separate triggers.

Why LangGraph instead of just calling functions directly?
LangGraph gives us a structured, inspectable pipeline where each step is isolated, testable, and replaceable. If the evaluation logic needs to change, we update one node. The rest of the graph is untouched.

Graph 1 — Campaign Dispatch

Triggered by: POST /api/campaigns/{id}/trigger

START → load_company_and_leads → dispatch_calls → END

State: CampaignState

{
  "company_id": "6a377fa...",     // input — passed in by the router
  "company": {},                   // filled by load_company_and_leads
  "pending_leads": [],             // filled by load_company_and_leads
  "initiated": [],                 // filled by dispatch_calls
  "errors": []                     // filled by dispatch_calls (per-lead errors)
}

Node: load_company_and_leads

Queries MongoDB for the company document and all leads where status == "PENDING" and company_id matches. Converts ObjectIds to strings (JSON-safe). Returns both in the state.

Node: dispatch_calls

Loops over pending_leads. For each: calls vapi_service.initiate_outbound_call(), updates the lead to CALL_INITIATED, creates a call_log document, and stores the Vapi call ID. If one lead fails, its error is added to state["errors"] and the loop continues. This is the key resilience behaviour.

Graph 2 — Call Evaluation

Triggered by: Vapi's webhook hitting POST /api/webhooks/vapi

START → load_lead_by_call → evaluate_transcript → persist_outcome → END

State: EvaluationState

{
  "vapi_call_id": "019ee8d9-...", // from Vapi webhook payload
  "transcript": "AI: Hi Alice...", // from Vapi webhook payload
  "summary": "Customer wants...",  // from Vapi webhook payload
  "lead": {},                       // filled by load_lead_by_call
  "company": {},                    // filled by load_lead_by_call
  "verdict": "QUALIFIED",          // filled by evaluate_transcript
  "reasoning": "Customer stated..." // filled by evaluate_transcript
}

Node: load_lead_by_call

Finds the lead document where vapi_call_id matches. Also loads the company so the LLM has context about what this company sells. If no lead is found, raises a ValueError which becomes a 404 HTTP response.

Node: evaluate_transcript

The brain of the system. Builds a prompt that includes the company context and the full transcript, then asks the LLM to classify the outcome. Returns a JSON verdict. Uses OpenAI first, Gemini as fallback, and NEEDS_REVIEW if both fail or return unparseable output.

Node: persist_outcome

Writes the verdict back to two places: updates leads.status, and fills in the call_logs document with the transcript, summary, and LLM reasoning.

Voice Layer

Vapi Integration

Vapi handles the entire telephony layer — placing the call, running the AI conversation, transcribing it, and posting the results back. We just tell it what to say and where to report.

Initiating an outbound call

We call POST https://api.vapi.ai/call/phone with a payload that includes the phone number, the customer's name, and a fully assembled assistant configuration.

// What we send to Vapi for each lead
{
  "phoneNumberId": "d19b97e9-...",  // our "from" number on Vapi
  "customer": {
    "name": "Alice Johnson",
    "number": "+14085550101"
  },
  "assistant": {
    "firstMessage": "Hi Alice, this is an assistant from Sunset Realty...",
    "transcriber": { "provider": "deepgram", "model": "nova-2" },
    "model": {
      "provider": "openai",
      "model": "gpt-4o-mini",
      "messages": [{ "role": "system", "content": "<dynamic system prompt>" }]
    },
    "voice": { "provider": "11labs", "voiceId": "21m00Tcm4TlvDq8ikWAM" },
    "serverUrl": "https://your-ngrok-url.ngrok.io/api/webhooks/vapi"
  }
}

Dynamic Prompting (Bonus Feature)

Each company has a prompt_instructions field. The _build_system_prompt() function combines it with the customer's name to create a unique system prompt per call. Sunset Realty's AI talks about house-buying. CityNest's AI talks about renting apartments. Same infrastructure, completely different conversations. 

def _build_system_prompt(lead, company):
    return (
        # Company's custom instructions come first
        f"{company['prompt_instructions']}\n\n"
        # Then we personalize with the actual customer name
        f"You are currently speaking with {lead['name']}. "
        "Keep the call under 3 minutes. "
        "At the end, summarize what the customer said about their intent."
    )

What Vapi returns

Vapi responds immediately with a call object. We save the id field as vapi_call_id on the lead — this is what we use later to match the webhook back to the right lead.

Free tier limitation: Free Vapi phone numbers are US numbers and cannot place calls to international numbers (+91 India, etc.). For testing with an Indian number, you need to purchase an Indian number in the Vapi dashboard (~$2-3/month).
Inbound Events

Webhook Handler

When a call ends, Vapi makes an HTTP POST to our server with the full call report. This is where the evaluation pipeline starts.

What Vapi sends

// Vapi end-of-call-report webhook payload
{
  "message": {
    "type": "end-of-call-report",
    "call": {
      "id": "019ee8d9-cb3a-700f-9d7a-f162bd945613",
      "customer": { "number": "+14085550101" }
    },
    "transcript": "AI: Hi Alice...\nAlice: Hi, yes I'm interested in buying...",
    "summary": "The caller expressed interest in purchasing a 3-bedroom home."
  }
}

What our handler does with it

📥 Webhook arrives
1
Read raw bytes first (not request.json()) so we can log the payload and handle empty bodies gracefully — Vapi sends empty pings to verify the endpoint is alive.
2
Parse as JSON. Extract message.type. If it's not end-of-call-report, return { "status": "ignored" } — we don't care about mid-call status updates.
3
Extract call.id, transcript, and summary. Pass them into evaluation_graph.ainvoke().
4
The graph runs three nodes (load → evaluate → persist). Return the verdict to Vapi as the HTTP response.
Public URL requirement: Vapi is a cloud service. Your server must have a publicly accessible HTTPS URL. Locally, use ngrok http 8000 and set WEBHOOK_BASE_URL in .env to the ngrok URL. In production, it's your Cloud Run URL automatically.
AI Classification

LLM Transcript Evaluation

The LLM is given the transcript and asked to classify the outcome. It is NOT hardcoded logic — the LLM reads the actual conversation and makes a judgment.

The prompt structure

You are analyzing a completed real estate sales call transcript.

Company: Sunset Realty
Company Focus: You work for Sunset Realty, a premium home-buying agency...

Call Transcript:
AI: Hi Alice, this is an assistant from Sunset Realty...
Alice: Yes hi, I've been looking at the market actually...

Classify the lead into exactly one of these categories:
- QUALIFIED: clear interest (wants to buy/sell/rent, asked questions, gave details)
- NOT_INTERESTED: clearly declined or is not interested
- NEEDS_REVIEW: too short, confusing, or you are not confident

Return ONLY valid JSON:
{
    "verdict": "QUALIFIED",
    "reasoning": "one sentence explanation"
}

The fallback chain

Try OpenAI GPT-4o-mini → success? → use result
↓ failed
Try Google Gemini 1.5 Flash → success? → use result
↓ failed
Set NEEDS_REVIEW → human follows up

OpenAI uses response_format: { type: "json_object" } which forces the model to return valid JSON — no parsing failures. Gemini doesn't support this so we catch JSON errors and fall back to NEEDS_REVIEW for those too.

Module Reference

database/

connection.py

Manages one shared MongoDB client for the lifetime of the application.

connect()async

Creates the MongoDB client and pings the server to verify connectivity. Called once in main.py's @app.on_event("startup"). If this fails, the app won't start — which is intentional.

Returns a reference to the database. Called by every node and router that needs to read/write data. Because MongoDB connections are thread-safe and reused by Motor, this is efficient even under high load.

Cleanly closes the MongoDB connection on app shutdown. Prevents connection leaks.

models.py

Pydantic models used as request/response bodies in the routers. The LeadStatus enum is the single source of truth for all valid status values.

seed.py

Creates 2 companies and 6 leads the first time the server starts. Checks for existing data first — completely safe to restart.

run_seed()async

Checks if any companies exist. If yes, prints "skipping" and returns immediately. If no, inserts the two companies and attaches 3 leads each. The leads use the first company's _id for leads 0-2, second company for leads 3-5.

Module Reference

services/vapi_service.py

All Vapi REST API communication lives here. Other parts of the system never touch the Vapi URL directly.

initiate_outbound_call()async (lead, company) → dict

The main function. Calls Vapi's /call/phone endpoint. Builds the assistant config inline (no pre-created assistant needed) using the company's profile for the system prompt. Logs the full Vapi error response to the terminal if it fails — so you can see exactly what Vapi rejected.

leadLead document from MongoDB (must have name, phone, _id).
companyCompany document (must have name, prompt_instructions).
_build_system_prompt()(lead, company) → str

Private helper. Combines the company's custom instructions with the lead's name. This is the "Dynamic Prompting" bonus requirement — every call sounds specific to the company making it.

get_call_details()async (vapi_call_id) → dict

Fetches full call details from Vapi's /call/{id} endpoint. Useful for debugging when the webhook data seems incomplete.

Module Reference

agent/

state.py

Defines the TypedDict schemas for both graphs. TypedDicts are just dictionaries with type hints — they tell LangGraph what shape each node's input and output should be.

nodes.py — Campaign nodes

load_company_and_leads()async (CampaignState) → CampaignState

First node. Fetches the company by company_id from MongoDB, then queries all leads with status == "PENDING" for that company. Converts ObjectIds to strings (MongoDB ObjectIds aren't JSON-serializable). Returns updated state with company and pending_leads filled.

dispatch_calls()async (CampaignState) → CampaignState

Second node. Loops through pending_leads. For each, calls initiate_outbound_call(). On success: updates lead to CALL_INITIATED, creates a call_log, appends to initiated. On failure (any exception): updates lead to FAILED, appends error to errors, continues loop. This try/except per-lead is critical — one bad number shouldn't stop the whole campaign.

nodes.py — Evaluation nodes

load_lead_by_call()async (EvaluationState) → EvaluationState

Looks up the lead using vapi_call_id — the ID we saved when we initiated the call. Also loads the company so the LLM evaluation prompt can include company context. Raises ValueError if not found (becomes 404).

evaluate_transcript()async (EvaluationState) → EvaluationState

Builds the classification prompt, tries OpenAI first, Gemini second, NEEDS_REVIEW if both fail. Validates the verdict is one of three known values — guards against the model returning unexpected text. Empty transcript → immediately returns FAILED.

persist_outcome()async (EvaluationState) → EvaluationState

Two writes: leads.update_one() to set the new status, and call_logs.update_one() to fill in the transcript, summary, and reasoning. Uses $set so other fields aren't touched.

graph.py

Assembles both graphs and compiles them at import time. Compiled graphs are module-level singletons — they're created once and reused for every request.

build_campaign_graph()→ CompiledGraph

Creates a StateGraph(CampaignState), adds two nodes, sets entry point to load_company_and_leads, adds edges, and compiles. The resulting object has an .ainvoke() method.

build_evaluation_graph()→ CompiledGraph

Same pattern. Three nodes, linear edges, compiled. Entry point is load_lead_by_call.

Module Reference

routers/

companies.py — GET /api/companies/

Standard CRUD. The _serialize() helper converts MongoDB's _id ObjectId to a plain string id field — because ObjectIds can't be JSON-serialized directly.

leads.py — GET /api/leads/

The ?company_id= query parameter is the multi-tenancy filter on the frontend. Passing no parameter returns all leads (admin view). The PATCH /status endpoint is a manual override for testing without making a real call.

campaigns.py — POST /api/campaigns/{id}/trigger

One-liner: builds the initial state and calls campaign_graph.ainvoke(). Returns a summary of what happened. Raises 404 if the company doesn't exist, 500 if the graph itself crashes.

webhooks.py — POST /api/webhooks/vapi

The public-facing endpoint Vapi calls. Reads raw bytes (not request.json()) to handle empty bodies without crashing. Logs every event type so you can see exactly what Vapi is sending. Only processes end-of-call-report events — everything else is logged and ignored.

Module Reference

main.py

The application entry point. Assembles the FastAPI app, configures middleware, registers routers, and handles startup/shutdown.

# startup: connect to DB, seed if empty
# shutdown: close DB connection
# CORS: open for all origins (development)
# Routers: companies, leads, campaigns, webhooks

The startup handler is why the DB connection is available by the time the first request arrives — connect() runs before the server starts accepting traffic.
